# SL-81 Server Hardening Stacked Delivery Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Deliver every SL-81 simulation, security, concurrency, operations, and deployment hardening requirement as six reviewable stacked pull requests.

**Architecture:** Keep one Linear issue as the scope source of truth while isolating each risk domain behind a dedicated branch and pull request. Every branch starts from the previous branch, preserves a green full tree, and exposes only the diff for its own stack to reviewers.

**Tech Stack:** Go 1.25, `net/http`, `log/slog`, `nhooyr.io/websocket`, Prometheus Go client, Bash, OpenAPI, AsyncAPI, Make, GitHub stacked pull requests.

## Global Constraints

- Linear issue `SL-81` is the source of truth; do not create additional implementation issues.
- Preserve the current `Step(inputs) -> Snapshot`, matchmaking Ready/countdown/start, GameEnd, and REST error contracts unless a stack explicitly changes and documents them.
- Do not add account auth, persistence, production matchmaking/rating, bot replacement, reconnect grace, respawn, score, dashboards, schedulers, runners, Kubernetes, horizontal scaling, or Cloudflare/DNS changes.
- Never expose a token, token digest, `Authorization` value, or complete query string in DTOs intended for other players, logs, metrics, test failure messages, or documentation examples.
- `/metrics` must remain absent from the public API listener and bind only to `127.0.0.1:9090` by default.
- Every REST/WebSocket contract change must update `api/openapi.yaml`, `api/asyncapi.yaml`, `ai-docs/api-reference.md`, `ai-docs/api-docs.md`, and the relevant protocol/architecture/decision documents in the same stack.
- Every stack must pass `make ci` before commit and before push; stacks 3-5 must also pass `go test -race ./internal/simulation ./internal/rooms ./cmd/server`.
- Use commit titles in the form `[SL-81] <semantic>(<scope>): <Korean summary>` followed by bullet points describing the changes.
- Use the repository pull request template and concise Korean `해요체` bullets.
- Do not mark SL-81 Done until all six pull requests are merged; opening the first PR moves it to In Review.
